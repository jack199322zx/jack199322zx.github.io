承接上回dubbo源码解析之consumer订阅

我们这次详细的看一下dubbo怎么实现RPC远程调用的过程：
上回说到，在ZK中拿到了provider的URL后，AbstractRegistry.notify() -> RegistryDirectory.refreshOverrideAndInvoker() -> toInvokers() ->
DubboProtocol.refer()，我们来看一下这个方法

DubboProtocol.initClient() -> Exchanges.connect()

可以看到，Exchanges.connect()方法创建了HeaderExchangeClient对象，而这个对象初始化的时候需要传入一个Client，而这个Client正是在Transporters.connect返回的。
我们来看一下这个方法：

getTransporter返回了NettyTransporter，其connect方法创建了一个NettyClient，看一下NettyClient做了什么：

在AbstractClient构造方法中调用子类的doOpen()，我们终于看到了Netty的影子，创建好了netty客户端，然后在doConnect方法中，建立连接。

这样我们就成功与provider端建立了SocketChannel，可以与之通信了。
那么建立完了连接之后，dubbo还做了什么呢，buildInvokerChain，将DubboProtocol返回的DubboInvoker传入，对DubboInvoker进行一个拉链操作，封装了一系列Filter。封装成MockClusterInvoker将其返回，最终返回一个动态代理对象。
然后在我们调用接口的方法时，代理对象将会执行invoke操作，经过一系列调用，指向buildInvokerChain中创建的内部类（Invoker），这个Invoker执行的操作如下：

调用过程ConsumerContextFilter.invoke() -> FutureFilter.invoke() -> MonitorFilter.invoke() -> AbstractInvoker.invoke() -> DubboInvoker.invoke() ->
HeaderExchangeChannel.request() -> NettyChannel.send()，于是就到了provider端的处理。

provider端的处理和consumer端其实也很相似，有兴趣的朋友可自行debug，嘿嘿。

至此，RPC调用过程就完结了。
